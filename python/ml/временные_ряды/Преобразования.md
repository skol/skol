# Что делает Box-Cox
**Box-Cox** — это **семейство преобразований**:
$$y(\lambda) = \begin{cases}
  \frac{y^\lambda -1}{\lambda}, & \lambda \neq 0 \\
  \ln(y),   & \lambda = 0
\end{cases}$$
**Ключевая особенность:** $\lambda$ (лямбда) **подбирается автоматически** для максимизации нормальности распределения.

# Логарифм — частный случай Box-Cox

При $λ = 0$ Box-Cox становится логарифмом. То есть **логарифм — это один из возможных вариантов** Box-Cox.
Но:
- Box-Cox может выбрать λ = 0.3, 0.5, -0.2 и т.д.    
- Логарифм — это **фиксированное** преобразование

# Принципиальные различия

| Характеристика | Логарифм                               | Box-Cox                                             |
| -------------- | -------------------------------------- | --------------------------------------------------- |
| Применимость   | Только для **положительных** данных    | Только для **положительных** данных                 |
| Интерпретация  | Естественная (проценты, отношения)     | Сложная (λ ≠ 0 — неинтуитивно)                      |
| Автоматизация  | Ручной выбор                           | Автоматический подбор λ                             |
| Риск           | Нет (кроме случаев, когда не подходит) | Переподбор (может выбрать λ, которая "не физична")  |
| Сезонность     | Стабилизирует дисперсию                | Стабилизирует дисперсию + нормализует распределение |

# Когда что выбирать
## Выбираем **логарифм**, если:

1. **Данные — это счетчики, деньги, население** — там логарифм имеет естественную интерпретацию (процентные изменения)    
2. **Нужна интерпретируемость** — после логарифма можно говорить "изменение на 1%"    
3. **Дисперсия растет пропорционально уровню** — это классический случай для log    
4. **В задаче важна простота** — коллеги/заказчики поймут логарифм, но не поймут Box-Cox с λ = 0.27
```python
# Когда log — очевидный выбор
df['log'] = np.log(df['Count'])
```
## Выбираем **Box-Cox**, если:
1. **Логарифм не помог** — дисперсия все еще нестабильна или распределение далеко от нормального    
2. **Нужна максимальная нормальность** для последующего моделирования (например, для некоторых моделей)    
3. **Нет естественной интерпретации** — можно позволить себе "черный ящик"    
4. **Экспериментальный режим** — хотим посмотреть, улучшит ли автоматический подбор λ качество
```python
from scipy.stats import boxcox

# Box-Cox автоматически подберет lambda
df['boxcox'], lam = boxcox(df['Count'])
print(f'Подобранная lambda: {lam:.4f}')
```
## Важный нюанс: Box-Cox только для положительных данных

**Оба преобразования требуют, чтобы все значения были > 0.**
Если в данных есть нули или отрицательные значения:
- Логарифм — ошибка    
- Box-Cox — тоже ошибка
Для таких случаев используют:
- **log(x + c)** — сдвиг на константу    
- **Yeo-Johnson** — аналог Box-Cox, но работает с отрицательными значениями
## Что говорят практики
В реальной работе с временными рядами:

|Специалист|Предпочтение|
|---|---|
|**Аналитик/бизнес**|Логарифм (понятно, интерпретируемо)|
|**Статистик/эконометрист**|Box-Cox (математически обоснованно)|
|**ML-инженер**|Часто вообще не преобразует, использует модели, устойчивые к масштабу|
|**Продакшн**|Логарифм (проще деплоить, меньше сюрпризов)|

Преобразования применяются **последовательно, пока не уберем все источники нестационарности** (тренд, дисперсию, сезонности на разных периодах).
# Почему может потребоваться больше двух преобразований

Пример про **месячную + квартальную + годовую сезонность** — это классический случай, когда одного `diff(12)` может быть недостаточно.
## Давай разберем на пальцах
Представь, что у тебя **ежемесячные данные** с двумя сезонностями:
- **Годовая сезонность (12)** — очевидна    
- **Квартальная сезонность (3)** — например, налоговые периоды, отчетности  

**Что будет на ACF после `log` и `diff(1)` (убран тренд):**
- Пики на лагах 12, 24, 36... (годовая)    
- **И** пики на лагах 3, 6, 9... (квартальная, вложенная внутрь годовой)    

**Что делаем:**
1. `log` — стабилизация дисперсии    
2. `diff(1)` — убираем тренд    
3. `diff(12)` — убираем **годовую** сезонность    
4. **Смотрим ACF снова** — если остались пики на лагах 3, 6, 9 → берем еще `diff(3)`    
5. Итог: `log(value).diff(1).diff(12).diff(3)`
## Пример из реальной жизни (не учебный)
**Почасовые данные энергопотребления:**
- Суточная сезонность (24)    
- Недельная сезонность (168 = 24×7)    
- Годовая сезонность (8760 = 24×365)    

Типичные преобразования:
```python
df['stationary'] = np.log(df['value']).diff(1).diff(24).diff(168)
# (если еще и тренд убирать нужен)
```
## Важное ограничение: "замучивание"

**Не любое количество преобразований оправдано**. Есть признаки, что вы переборщили:

| Признак                                                          | Что означает                         |
| ---------------------------------------------------------------- | ------------------------------------ |
| Дисперсия **выросла** после преобразования                       | Overdifferencing                     |
| ACF на лаге 1 стала **отрицательной и большой** (например, -0.6) | Взяли лишнюю разность                |
| Потеряли > 20-30% наблюдений                                     | Особенно критично для коротких рядов |
| Не можете **интерпретировать**, зачем сделали преобразование     | Скорее всего, оно лишнее             |
## Когда может понадобиться diff(3) для месячных
Реальные кейсы:
- **Бизнес-циклы:** квартальные отчеты влияют на показатели (например, продажи перед квартальным закрытием)    
- **Сельское хозяйство:** посевная (весна), уборочная (осень) — но это скорее лаги 3-6, а не жесткая сезонность    
- **Туризм:** высокий сезон (лето) vs низкий (зима) — разница в 6 месяцев, не 3
## Резюме
- **Да**, преобразований может быть несколько (2, 3, иногда 4)    
- **Порядок** имеет значение: стабилизация дисперсии → тренд → большая сезонность → малая сезонность    
- **Ориентир** — ACF/PACF после каждого шага    
- **Стоп-сигнал** — признаки overdifferencing

# Выравнивание
Данные ряда нужно выравнивать, но не "по границе", а **по полноте сезонного цикла**.
## В чем проблема
Когда вы берем сезонную разность `diff(12)`, мы вычитаем:
```text
y[t] - y[t-12]
```
Если ряд **начинается не с января** или **заканчивается не в декабре**, то:
- Для первых 12 наблюдений у вас нет пары (y[t-12] не существует) → они пропадают    
- Это нормально и неизбежно    
- Но есть нюанс с **интерпретацией сезонной компоненты**
## Что значит "выравнивать по границе"
### Случай 1: Ряд начинается не с января (например, с апреля)
**Проблема:** если просто береv `diff(12)`, мы сравниваем:
- Апрель 1965 с апрелем 1964 (нет данных) → NaN    
- Май 1965 с маем 1964 → NaN    
- ...    
- Декабрь 1965 с декабрем 1964 → NaN    
- Январь 1966 с январем 1965 → ОК
**Результат:** теряем первые 12 наблюдений, но это **нормально**.
**Ничего дополнительно "выравнивать" не нужно**, потому что:
- Сезонная разность работает с парой "месяц-к-месяцу"    
- Не важно, с какого месяца начинается ряд — важно, чтобы у вас были пары для каждого месяца
### Случай 2: Ряд заканчивается не в декабре (например, в ноябре)
**Проблема:** последнее значение (ноябрь 1971) не имеет пары для декабря 1971? Нет, пара — это ноябрь 1970, который есть.
**Никакой проблемы нет.** Просто не сможем посчитать `diff(12)` для последних 12 наблюдений? Нет, посчитаем для всех, кроме первых 12.
## Реальная проблема: неполный сезонный цикл
Проблема возникает, когда **данных меньше, чем один полный сезонный цикл**:
```python
# Плохо: данные только за 8 месяцев
dates = pd.date_range('2023-01-01', periods=8, freq='M')
values = np.random.randn(8)
# Нельзя брать diff(12) — нет пары для большинства точек
```
## Еще один нюанс: сезонная декомпозиция

Если вы используете `seasonal_decompose` (а не `diff(12)`), там действительно важно, чтобы ряд **покрывал целое число сезонных периодов**:
```python
# seasonal_decompose требует, чтобы len(df) было кратно period
from statsmodels.tsa.seasonal import seasonal_decompose

# Если у вас 70 месяцев, а period=12 → 70/12 не целое число
# Может быть ошибка или неполная декомпозиция
decomp = seasonal_decompose(df['Count'], model='multiplicative', period=12)
# Лучше обрезать ряд до 60 или 72 месяцев
```
**Но:** для `diff(12)` это **не требуется**. `diff()` работает с любым количеством данных.
## Когда нужно выравнивание на практике

В реальных проектах выравнивание нужно, когда:
- Строим **сезонные индексы** для прогнозирования    
- Вы используем **скользящие окна** (rolling statistics)    
- Сравниваем **сезонно-скорректированные** ряды с официальной статистикой (там часто требуют полные годы)
# Сравнение `df.diff(periods=12)` и `df[12:] - df[:-12].values`
В лекции упоминалась df.diff, и рекомендовалось ручное вычисление разности.
```python
import pandas as pd
import numpy as np

# Создаем тестовый ряд
df = pd.DataFrame({'value': np.arange(1, 25)}, 
                   index=pd.date_range('2023-01-01', periods=24, freq='M'))

# Вариант 1: diff(12)
diff1 = df['value'].diff(12)

# Вариант 2: ручное вычитание
diff2 = df['value'][12:] - df['value'][:-12].values

# Сравниваем
print(diff1[12:].equals(diff2))  # True
```
**Результат:** идентичны.

## Но! Важный нюанс с индексом
```python
# diff(12) сохраняет индекс
print(diff1.index[12:])  # 2024-01-01, 2024-02-01...

# Ручное вычитание без .values может сбить индекс
diff2_wrong = df['value'][12:] - df['value'][:-12]
# Здесь pandas попытается выровнять по индексу и выдаст NaN
```
## Когда может быть разница

### 1. В numpy (без учета индексов)
```python
import numpy as np

arr = np.array([1, 2, 3, 4, 5, 6])
# diff с периодом 2
diff_np = np.diff(arr, n=2)  # array([2, 2]) — совсем другая логика!

# В numpy нет diff(periods=12) как в pandas
# np.diff(arr, n=2) — это вторая разность, не сезонная!
```
### 2. В R
В R `diff(series, lag=12)` — это именно сезонная разность, работает как в pandas.
### 3. Если забыть про выравнивание индексов
```python
# Так делать НЕЛЬЗЯ
wrong = df['value'][12:] - df['value'][:-12]  # индекс не совпадет → NaN

# Правильно
correct = df['value'][12:].values - df['value'][:-12].values
```
# Почему dropna() обязателен
После `diff()` у вас появляются **NaN** в начале ряда:
```python
# После diff(1) — первый элемент NaN
# После diff(12) — первые 12 элементов NaN
# После комбинации — еще больше NaN
```
**Если не сделать dropna():**
- Тест Дики-Фуллера (ADF) упадет с ошибкой    
- ACF/PACF не построятся    
- Многие функции statsmodels просто откажутся работать
## Как правильно
```python
# После преобразований
df['stationary'] = np.log(df['Count']).diff().diff(12)

# Убираем NaN перед тестами и графиками
df_clean = df['stationary'].dropna()

# Теперь можно
adf_test(df_clean)
plot_acf(df_clean)
```
## Важный нюанс: не удаляйте из исходного df
Исходный df оставляем нетронутым, чтобы:
- Можно было вернуться и проверить шаги    
- Не потерять информацию о том, сколько наблюдений было потеряно
## Сколько наблюдений вы потеряете
Для тренировочного датасета (77 месяцев):

| Преобразования     | Потеряно | Осталось |
| ------------------ | -------- | -------- |
| Исходный           | 0        | 77       |
| + log (нет потерь) | 0        | 77       |
| + diff(1)          | 1        | 76       |
| + diff(12)         | еще 12   | 64       |
| **Итого**          | **13**   | **64**   |
Это **нормально**. В отчете можно указать: "после преобразований потеряно 13 начальных наблюдений".
